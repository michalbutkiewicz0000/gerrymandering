0. CEL I ZASADA ODBIORU
======================

Celem pozostaje kompletny program, który:

1. rekonstruuje poligony polskich obwodów głosowania z rejestru PKW i PRG,
2. buduje prawidłowy graf sąsiedztwa obwodów,
3. importuje wyniki i scenariusze wyborcze,
4. sprawdza reguły prawa wyborczego,
5. dokładnie optymalizuje spójne podziały na okręgi dla komitetu lub kandydata,
6. działa przez Docker, API, CLI i interfejs WWW,
7. posiada testy, certyfikaty optimum oraz udokumentowane bramki odbioru.

Nie wolno uznać projektu za skończony tylko dlatego, że testy jednostkowe są
zielone. Odbiór wymaga dowodów z docs/acceptance.md oraz kontroli każdego
pozostałego punktu opisanego niżej.

3. GRAF KRAJOWY — NAPRAWA TOPOLOGII WYKONANA, WYMAGA BRAMEK
===========================================================

Artefakt:

    data/processed/snapshots/49b45322-b35c-4e6e-8dd7-ccfe498d3e00/graph.json

Pierwszy wynik bez tolerancji topologicznej miał 71106 krawędzi, 207
składowych i 2 izolaty. Diagnoza wszystkich par między składowymi wykazała:

    pary między składowymi w odległości <= 1 cm = 8677
    liczba składowych po uwzględnieniu tych par = 1

Kontrola rzeczywistych par wykazała, że po obu stronach granicy gminy PRG
istnieją równoległe odcinki długości setek lub tysięcy metrów, lecz zapisane
osobno geometrie różnią się o mikroskopijny ułamek metra. Przykłady:

    142202_3 — 281102_2: około 5908 m równoległej granicy
    142106_14 — 146517_666: około 592 m równoległej granicy

Nie były to odległe województwa ani dowolne najbliższe poligony: Chorzele,
Janowo i Wielbark leżą przy granicy mazowiecko-warmińsko-mazurskiej.

Zaimplementowano w `build_adjacency` jawnie konfigurowalną tolerancję
`boundary_tolerance_m`, domyślnie 0,01 m. Algorytm mierzy długość części obu
granic znajdujących się w pasie tolerancji i bierze mniejszy z dwóch pomiarów.
Nadal obowiązuje minimalny wspólny odcinek 1 m, dlatego kontakt tylko w punkcie
nie tworzy krawędzi. Tolerancję można wyłączyć wartością 0.

Parametr dodano do CLI i API. Artefakt zapisuje teraz reprodukowalne
`build_parameters`: key_column, metric_crs, min_shared_border_m i
boundary_tolerance_m. Dodano regresje mikroszczeliny i wyłączonej tolerancji;
dotychczasowy test kontaktu narożnego nadal przechodzi.

Graf kraju zbudowano ponownie. Aktualny wynik:

5. DOCKER I OGRANICZENIE TEMPERATURY
====================================

Użytkownik zgłosił zbyt wysoką temperaturę rdzeni. Znaleziono wcześniej dwa
równoległe, stare przebiegi rekonstrukcji i zakończono je. Obecnie pełna
rekonstrukcja nie działa. docker-compose.yml ma niezatwierdzone limity:

    db     1 CPU
    api    1 CPU
    worker 4 CPU

Kontrola `docker inspect` potwierdziła odpowiednio NanoCPUs:

    1000000000
    1000000000
    4000000000

Nie używać `--workers 20` ani `--workers 0`. Domyślnie używać `--workers 4`,
a jeśli temperatury nadal są wysokie — `--workers 2`. Nie uruchamiać ciężkich
zadań jednocześnie z Docker build albo pełnym pytest.

Po zakończeniu zmian wykonać bramkę obrazu:

    docker compose up -d --build
    docker compose ps
    docker compose exec -T worker gerry migrate
    docker compose exec -T worker gerry doctor
    docker compose exec -T worker gerry solver-smoke
    curl -fsS http://localhost:8000/ready

Aktualizacja 15 lipca 2026 r.: `docker compose build` po wszystkich opisanych
zmianach zakończył się sukcesem. Kompilacja SCIP/SoPlex/VIPR była w cache,
instalacja aplikacji przeszła, wykryto `viprcomp` i `viprchk`, a build wykonał
`Zweryfikowano 3 akty prawne`. Próba uruchomienia poniższej bramki runtime
została odrzucona przez limit narzędzia zatwierdzającego dostęp do demona
Docker, nie przez błąd obrazu:

    docker compose up -d --force-recreate
    docker compose exec -T worker gerry migrate
    docker compose exec -T worker gerry doctor
    docker compose exec -T worker gerry solver-smoke
    curl -fsS http://localhost:8000/ready

Należy wykonać te komendy bezpośrednio w terminalu użytkownika i wkleić wynik
albo powtórzyć je, gdy dostęp narzędzia do Dockera będzie ponownie możliwy.

Oczekiwane kluczowe wyniki:

    Schemat bazy jest aktualny.
    Solver exact: OK (SCIP 10.0.2 EXACTSOLVE)
    VIPR: OK (viprcomp=tak, viprchk=tak)
    Status: OPTIMAL; certyfikat: zweryfikowany
    Brak rozwiązania: INFEASIBLE; certyfikat: zweryfikowany

====================================

Po pełnych bramkach grafu wykonać audyt wymagań z docs/acceptance.md, README.md oraz
docs/architecture.md. Nie wystarczy założyć, że starsze wpisy są aktualne.

1. pełna bramka testów/Dockera dla końcowej topologii grafu kraju,
2. pełna synchronizacja krajowej migawki i grafu do PostGIS,

5. niezależna kontrola profili prawnych przez eksperta wyborczego.

Punktu 5 nie wolno samodzielnie oznaczyć jako wykonany bez rzeczywistej opinii
eksperta. Program może być technicznie ukończony, ale status pełnej walidacji
prawnej musi wtedy pozostać jawnie oczekujący.

Dla każdej pozycji matrycy odbioru zapisać dowód: plik/test, komendę, wynik i
datę. Dopiero gdy wszystkie wymagania celu mają bezpośredni dowód i nic nie
pozostaje, można uznać cały projekt za ukończony.
